## 📌 프로젝트 소개

이 프로젝트는 **그룹웨어의 핵심 도메인(인사, 결재, 소통) 토이 프로젝트**입니다. 

실제 사내 시스템에서 발생할 수 있는 데이터 정합성 문제, 트래픽에 따른 조회 성능 저하, 다중 기업(SaaS) 환경에서의 보안 격리 등의 현실적인 문제들을 정의하고, 이를 백엔드 아키텍처로 꼼꼼하게 해결해 나가는 과정에 초점을 맞추었습니다.

* 🔙 **Backend**: 무작정 많은 기능을 붙이기보다 핵심 비즈니스 로직, 데이터 격리, 동시성 제어 및 병목 지점 개선에 온전히 집중하여 깊이 있게 구현했습니다.
* 🎨 **Frontend**: 유저가 실제로 시스템을 조작해 볼 수 있는 최소 기능(MVP) 수준으로 유지하기 위해, AI 도구(Claude 등)를 적극 활용하여 생산성을 극대화하는 방식으로 전임하여 개발했습니다.

## 🚀 Backend Technical Challenges

백엔드 개발자로서 단순히 기능 구현에 그치지 않고, 대규모 트래픽과 기업용(B2B) 도메인에서 발생할 수 있는 문제들을 정의하고 해결했습니다.

<br />

### 🛡️ 1. Spring Boot Filter를 활용한 SaaS 멀티 테넌시 데이터 격리
- **문제 정의**: 기업 기밀이 포함된 B2B 서비스 특성상 타 기업의 데이터가 노출되는 것은 치명적입니다. 비즈니스 로직에 수동으로 기업 조건을 넣는 것은 휴먼 에러의 위험이 큽니다.
- **해결 방안**: JWT 토큰에 담긴 `company_id`를 Spring Filter 영역에서 가로채 context에 저장했습니다. 이후 JPA Filter 또는 Hibernate TenantResolver를 연동하여 개발자가 별도 조건을 명시하지 않아도 **자동으로 해당 기업의 데이터만 조회·수정되는 안전한 격리 계층**을 구현했습니다.

<br />

### ⚖️ 2. 전자결재(휴가/지출) 동시성 제어 및 대용량 파일 업로드
- **문제 정의**: 한 부서의 남은 연차가 1일일 때 두 사원이 동시에 신청하는 경우 자원 정합성이 깨질 수 있습니다. 또한 채팅이나 게시판의 대용량 파일이 서버 메모리를 타게 되면 부하가 심해집니다.
- **해결 방안**: 
  - 정합성이 중요한 결재 도메인에는 DB Lock(Pessimistic Lock 등)을 활용해 **동시 요청 시에도 데이터가 꼬이지 않도록 제어**했습니다.
  - 대용량 파일은 백엔드 서버를 거치지 않고 S3로부터 `Presigned URL`을 발급받아 클라이언트가 직접 업로드하게 하여 **서버의 네트워크 및 메모리 부하를 획기적으로 절감**했습니다.

<br />

### ⚡ 3. Redis Bitmap을 활용한 전사 근태 현황 초고속 조회
- **문제 정의**: 메인 대시보드에서 전 직원의 당일 출근 현황을 매번 RDBMS의 `COUNT` 쿼리로 조회하는 것은 기업 규모가 커질수록 DB에 큰 부하를 줍니다.
- **해결 방안**: Redis의 초경량 비트 연산 구조인 `Bitmap`을 채택했습니다. 사원 고유 번호를 Offset으로 두고 출근 시 bit를 `1`로 켜서, 수만 명의 출근 여부를 단 몇 KB의 메모리만으로 기록하고 `BITCOUNT`를 통해 **압도적인 속도로 실시간 출근 인원을 집계**했습니다.

<br />

### 💬 4. Redis Sorted Set 기반의 '안 읽은 메시지' 카운트 실시간 연산
- **문제 정의**: 사용자가 속한 수많은 채팅방의 안 읽은 메시지 수를 RDBMS 조회로 실시간 계산하는 것은 심각한 Read 부하를 야기합니다.
- **해결 방안**: Redis의 `Sorted Set`을 활용하여 각 채팅방의 최신 메시지 ID와 유저가 마지막으로 읽은 메시지 ID를 Score로 기록했습니다. 인메모리 연산을 통해 사용자가 접속할 때마다 **각 방의 안 읽은 카운트를 딜레이 없이 실시간으로 계산하여 반환**하는 고성능 아키텍처를 설계했습니다.
